# 移行時の注意点（過去の教訓まとめ）

現行サイト（Gware CMS, PHP系・動的）から新サイト（Next.js）への移行にあたり、
過去のトラブルから得た教訓を整理する。

## 1. サーバー脆弱性経由とみられるスパム注入被害

- 現行サイトで、サーバー側の脆弱性を突かれたとみられる経路で
  カジノ系スパムがページに注入された実績がある。
- **移行時の対策方針:**
  - 新サイトは静的生成（SSG）を基本とし、動的な管理画面・アップロード機能など
    攻撃対象になりうる仕組みは新CMS選定時に慎重に検討する。
  - CMS API連携時（`lib/news.js` を差し替える段階）は、取得したデータを
    そのままHTMLとして出力しない（XSS対策）。Reactの標準のレンダリングは
    デフォルトでエスケープされるため、`dangerouslySetInnerHTML` の使用は
    原則禁止とし、使う場合はサニタイズ処理を必須にする。
  - 旧サイトに残存する不審なコンテンツ・アカウント・プラグインを
    移行時にすべて棚卸しし、新サイトには一切引き継がない。

## 2. マージコンフリクトマーカーの本番混入

- 過去に `<<<<<<<` / `=======` / `>>>>>>>` などのGitマージコンフリクトマーカーが
  解消されないままコミットされ、本番ページに表示された実績がある。
- **移行時の対策:**
  - `.github/workflows/check-conflict-markers.yml` でCI上マーカーを検出し、
    マージ前に必ず失敗させる仕組みを導入済み（詳細は当該ファイル参照）。
  - レビュー時、差分にマーカーが残っていないか目視確認も引き続き行う。

## 3. その他の技術的負債（移行時に注意すること）

- 現行はPHPベースの自社開発CMSで、ページ構成やロジックがブラックボックス化
  している可能性がある。移行前に既存ページの挙動（フォーム送信先、
  問い合わせ処理の仕組みなど）を棚卸しすること。
- News/Recruitなど更新頻度の高いページは、担当者が非エンジニアでも
  更新できる仕組み（CMS連携）を優先的に検討する。
